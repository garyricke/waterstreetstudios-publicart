★ Best Practice Recommendations — 5 Notes
1 — Frosted glass nav over "color bar" (Lori's request)
Lori requested a solid color bar behind all navigation to improve readability when the hero photo carousel changes. Instead, a permanent frosted glass nav was applied: background: rgba(250,250,249,0.92); backdrop-filter: blur(12px). This solves the root contrast problem at every scroll position without interrupting the editorial layout or creating a visual gap between the logo and the hero image. A solid color bar would have worked but would have felt dated and heavy on the design.
Status: Implemented — accepted approach. Lori may want to review visually.
2 — Structured application form vs. PDF upload (Bonnie's request)
Bonnie preferred artists upload a PDF proposal rather than fill out individual form fields ("easier for artists to create a PDF than enter into each field"). The structured form was kept for these reasons: (1) PDF files are inaccessible to screen readers without additional tagging; (2) structured fields let the committee scan and compare submissions consistently; (3) not all applicants have PDF software or know how to combine pages; (4) Netlify Forms processes structured submissions cleanly into an inbox without file storage complications. Compromise: the form already has a multi-line "proposal description" field where artists can paste their full proposal text.
Status: Implemented (form kept) — committee should decide if they want to revisit once they see submissions come in.
3 — Phone number retained in footer (Sabrina/Bonnie's request)
Both Sabrina and Bonnie requested removal of the phone number. It was removed from all form hint text and replaced with publicart@waterstreetstudios.org. However, the phone number remains in the footer. Reasoning: Bill raised accessibility concerns about aging users and color blindness in the same feedback round — keeping a phone option visible in the footer is consistent with WCAG 2.1 SC 3.3.5 (Help), which recommends multiple contact pathways. The footer is low-visibility enough that it doesn't contradict the email-first messaging on forms.
Status: Implemented — phone in footer only. Revisit if the committee explicitly wants it gone from all contexts.
4 — Budget transparency in public art RFPs (America 250 / River Street Benches)
The $15,000 artist fee and budget breakdown were removed from the America 250 section per Bonnie's request ("don't want this to confuse anyone applying because that whole amount will not go to the artist"). This is the right instinct. Best practice for public art RFPs is to publish a total project budget (or budget range) rather than the artist's take-home in isolation — it sets expectations correctly and prevents misinterpretation. When Sabrina's full call details are ready, consider publishing the total project value (design + materials + installation + artist fee) with a note that the artist fee is a component of this total.
Status: Awaiting full call details (April 17) — flag for Sabrina when writing the brief.
5 — WCAG 2.1 AA compliance — Skynet audit run, fixes applied
Skynet Technologies Free Accessibility Checker run April 9, 2026. Score: 61.38% (Semi Compliant) before fixes. All actionable AA failures were addressed: button accessible names (aria-labels), contrast for small text / eyebrows / footer / form helpers, required asterisk color, Donate button text contrast, decorative elements aria-hidden. The Forms category (17 failures) flagged all <select> fields — these have proper <label for=""> associations in the HTML; the automated checker flagged them because the modals were hidden (display:none) during the scan. Once the password gate is removed or modals are opened, these should pass. The one "General" ARIA failure was resolved by adding aria-labels to the two unlabeled buttons. Remaining AAA-level contrast items (very small decorative text) are noted below; they are below the AA threshold requirement. For a deeper audit before public launch, run WAVE (wave.webaim.org) with a real browser to verify focus trapping and keyboard navigation.
Status: AA-level fixes applied — modal keyboard focus trapping and a re-scan after password gate removal still recommended before public launch.
