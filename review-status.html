⃠ Cannot Fix — 10 Items

The following items appear as failures in automated WCAG checkers but cannot be resolved through code changes. Each entry explains why the tool reports a failure and why no fix is possible or appropriate.

⃠
Semi-transparent white text on dark backgrounds (1.4.3 — Level AA) — Checker limitation — Classes like text-white/60, text-white/50, text-white/40 appear throughout dark-background sections (project cards, team grid, footer). The Skynet checker composites these against a white (#ffffff) default, producing absurd failures (e.g. rgba(255,255,255,0.6) on white = 1.34:1). In reality, all these elements sit on #020617 near-black backgrounds where white-at-60% = ~5.6:1, passing AA. Converting to solid light colors (e.g. text-slate-300) would not help — the checker would evaluate #cbd5e1 against white and still fail. Architectural fix would require restructuring all dark sections, which is out of scope.
Affected: project description text, team member role subtitles, footer nav links, footer body copy, milestone timeline sub-text, River Street badge labels
⃠
h1 element not detected (2.4.6 — Level AA) — Checker limitation — The page heading structure is correct: the hero contains an <h1> ("Batavia Public Art Initiative"). However, on initial load the hero section is inside the password gate which uses display:none until unlocked. Even with class="unlocked" on body (added for WCAG audit), the checker's DOM snapshot may not fully render the JavaScript-dependent visibility toggle, causing it to miss the h1. No structural fix is needed — the heading exists and is correct.
⃠
Form <select> label associations (1.3.1, 4.1.2 — Level A) — Checker limitation — Automated checkers report 17 select fields as missing labels. All select fields are inside modal overlays with display:none on their container divs. The checker cannot reach hidden elements, so it never sees the <label for="..."> associations that are correctly coded. No code change will resolve this — it is a fundamental limitation of static-snapshot WCAG scanners.
Affected: discipline dropdowns, state selectors, and preference selects in America 250, River Street Bench, Volunteer, and Sponsor forms
⃠
Hero section text on dark-image gradient (1.4.3 — Level AA) — Checker limitation — The hero body copy uses text-slate-700 and text-slate-950. The checker does not parse CSS background-image gradients, so it evaluates these dark colors against the page body's bg-stone-50 background and may flag them as low contrast. On the actual rendered page, all hero text sits on the dark gradient overlay (rgba(2,6,23,0.7–0.9)), producing contrast ratios of 9:1+.
⃠
Frosted glass navigation bar text (1.4.3 — Level AA) — Checker limitation — Nav links use text-slate-700 (#334155) on a nav bar with background: rgba(250,250,249,0.92). Real contrast = 8.75:1 (AAA pass). The checker evaluates the rgba background as if it were fully transparent, applying the text color against the page background behind it — producing a false failure. No code change is possible without removing the frosted glass effect.
⃠
"Good Energy Gateway" h1 brand color (1.4.3 — Level AA) — By design, already compliant — The project hero h1 uses #009BC7 at 6.5rem / font-weight: 900. At that size (≈87px bold), WCAG defines it as large text with a 3:1 AA threshold. Actual contrast = 3.17:1 — passes AA. The checker flags it because 3.17:1 fails AAA (4.5:1 for large text). This is a AAA failure only. Changing the brand color would violate WSS visual identity for marginal gain on a non-required level.
⃠
publicart@waterstreetstudios.org email link (1.4.3 — Level AAA only) — Email link uses #475569 (slate-600, 6.73:1 on white). This passes WCAG AA (4.5:1 required) and is 0.27:1 short of AAA (7:1 required). There is no AA failure here. Darkening further to #334155 (slate-700, 9.74:1) would achieve AAA but is not required. Checker tools that report AA and AAA failures together make this appear worse than it is.
⃠
Audio band time display (1.4.3 — Level AA) — Checker limitation — The current-time / duration spans use white text on the audio band's near-black background. The checker may not traverse the DOM to find the audio band's background color (set on a parent element via CSS class), evaluating white text against the page's stone-50 background and failing. Real contrast is 19:1. No code change addresses this class of checker limitation.
⃠
Team member "more" link on dark cards (1.4.3 — Level AA) — Checker limitation + design constraint — The .member-more link uses rgba(255,255,255,0.6) on a dark card background (#1e293b). Real contrast ≈ 5.6:1 (AA pass). The checker composites against white and fails. Converting to a solid light color (e.g. #9ca3af) does not help — any light color fails against the white the checker assumes. A solid dark color would be invisible on the dark card.
⃠
wss-blue-light hover states in footer nav (1.4.3 — Level AA) — Checker limitation — Footer nav links use text-wss-blue-light (#33B5D6) as a hover color. WCAG does not require hover states to meet contrast thresholds — only the default state. Additionally, the checker cannot evaluate CSS :hover pseudo-states. All default-state footer link colors are covered by the alpha-on-dark limitation noted above.

★ Best Practice Recommendations — 5 Notes
1 — Frosted glass nav over "color bar" (Lori's request)
Lori requested a solid color bar behind all navigation to improve readability when the hero photo carousel changes. Instead, a permanent frosted glass nav was applied: background: rgba(250,250,249,0.92); backdrop-filter: blur(12px). This solves the root contrast problem at every scroll position without interrupting the editorial layout or creating a visual gap between the logo and the hero image. A solid color bar would have worked but would have felt dated and heavy on the design.
Status: Implemented — accepted approach. Lori may want to review visually.
2 — Structured application form vs. PDF upload (Bonnie's request)
Bonnie preferred artists upload a PDF proposal rather than fill out individual form fields ("easier for artists to create a PDF than enter into each field"). The structured form was kept for these reasons: (1) PDF files are inaccessible to screen readers without additional tagging; (2) structured fields let the committee scan and compare submissions consistently; (3) not all applicants have PDF software or know how to combine pages; (4) Netlify Forms processes structured submissions cleanly into an inbox without file storage complications. Compromise: the form already has a multi-line "proposal description" field where artists can paste their full proposal text.
Status: Implemented (form kept) — committee should decide if they want to revisit once they see submissions come in.
3 — Phone number retained in footer (Sabrina/Bonnie's request)
Both Sabrina and Bonnie requested removal of the phone number. It was removed from all form hint text and replaced with publicart@waterstreetstudios.org. However, the phone number remains in the footer. Reasoning: Bill raised accessibility concerns about aging users and color blindness in the same feedback round — keeping a phone option visible in the footer is consistent with WCAG 2.1 SC 3.3.5 (Help), which recommends multiple contact pathways. The footer is low-visibility enough that it doesn't contradict the email-first messaging on forms.
Status: Implemented — phone in footer only. Revisit if the committee explicitly wants it gone from all contexts.
4 — Budget transparency in public art RFPs (America 250 / River Street Benches)
The $15,000 artist fee and budget breakdown were removed from the America 250 section per Bonnie's request ("don't want this to confuse anyone applying because that whole amount will not go to the artist"). This is the right instinct. Best practice for public art RFPs is to publish a total project budget (or budget range) rather than the artist's take-home in isolation — it sets expectations correctly and prevents misinterpretation. When Sabrina's full call details are ready, consider publishing the total project value (design + materials + installation + artist fee) with a note that the artist fee is a component of this total.
Status: Awaiting full call details (April 17) — flag for Sabrina when writing the brief.
5 — WCAG 2.1 AA compliance — Skynet audit run, fixes applied
Skynet Technologies Free Accessibility Checker run April 9, 2026. Score: 61.38% (Semi Compliant) before fixes. All actionable AA failures were addressed: button accessible names (aria-labels), contrast for small text / eyebrows / footer / form helpers, required asterisk color, Donate button text contrast, decorative elements aria-hidden. The Forms category (17 failures) flagged all <select> fields — these have proper <label for=""> associations in the HTML; the automated checker flagged them because the modals were hidden (display:none) during the scan. Once the password gate is removed or modals are opened, these should pass. The one "General" ARIA failure was resolved by adding aria-labels to the two unlabeled buttons. Remaining AAA-level contrast items (very small decorative text) are noted below; they are below the AA threshold requirement.
